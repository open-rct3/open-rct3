# Dumper OVL Viewer Plugin Guidance

This skill provides comprehensive guidance on building, configuring, and maintaining Extism WebAssembly plugins used to render OVL resource files in the OpenRCT3 Dumper content panel.

---

## 1. Plugin Architecture & Host-Side Discovery

The Dumper app integrates an Extism-powered plugin system that allows dynamically rendering raw OVL resource bytes as HTML in a `WebView2` (Windows) or `WKWebView` (macOS) panel.

### Discovery Paths
Plugins are compiled `.wasm` files. The host discovers them at startup from the following locations:
*   **Bundled**: `{app_dir}/plugins/*.wasm`
*   **User Data (Windows)**: `%APPDATA%/OpenRCT3/Plugins/*.wasm` followed by `%USERDOCUMENTS%/OpenRCT3/Plugins/*.wasm`
*   **User Data (macOS)**: `~/Library/Application Support/OpenRCT3/Plugins/*.wasm` followed by `~/Documents/OpenRCT3/Plugins/*.wasm`

### Host Interfaces
*   [IViewerPlugin](../../../Dumper/Plugins/IViewerPlugin.cs): Defines the plugin's metadata (`Name`, `Version`, `SupportedFileTypes`) and the `string Render(byte[] data)` method.
*   [ViewerPlugin](../../../Dumper/Plugins/ViewerPlugin.cs): Wraps an Extism `Plugin` instance, calling exported WASM methods inside a sandboxed runner with a default fuel limit of **50 units**.
*   [PluginManager](../../../Dumper/Plugins/PluginManager.cs): Scans discovery paths and maintains an ordered registry mapping `FileType` tag &rarr; viewer plugins. The first matching plugin is used as the default viewer.

## 2. WASM Plugin Contract

Every viewer plugin is written in AssemblyScript, compiles to a standalone WebAssembly module, and **must** export the following four functions:

| Export | Signature | Return Value | Purpose |
| :--- | :--- | :--- | :--- |
| `name` | `() → i32` | 0 | Outputs the display name, e.g., `"Integer Viewer"`, via `Host.outputString()` |
| `version` | `() → i32` | 0 | Outputs the semantic version, e.g., `"0.1.0"`, via `Host.outputString()` |
| `file_types` | `() → i32` | 0 | Outputs a JSON array of supported OVL tags, e.g., `'["int"]'`, via `Host.outputString()` |
| `render` | `() → i32` | 0 | Reads input bytes via `Host.input()`, processes them, and outputs HTML via `Host.outputString()` |

### Minimal Template (`index.ts`)
```typescript
import { Host } from "@extism/as-pdk";

export function name(): i32 {
  Host.outputString("Example Viewer");
  return 0;
}
export function version(): i32 {
  Host.outputString("0.1.0");
  return 0;
}
export function file_types(): i32 {
  Host.outputString('["xxx"]');
  return 0;
}
export function render(): i32 {
  let data = Host.input();
  let html = "<h1>Data: " + data.length.toString() + " bytes</h1>";
  Host.outputString(html);
  return 0;
}
```

## 3. Ground Truth: Plugin Implementations

The active ground truth implementation resides in the [plugins/](../../../plugins/) folder. All plugins leverage helper libraries located in [plugins/lib/](../../../plugins/lib/).

### Hex Viewer Utility
To guarantee consistent troubleshooting outputs, most plugins append a hex dump generated by [hexViewer.ts](../../../plugins/lib/hexViewer.ts):
*   Formats byte representations in a clean monospace HTML table.
*   Shows printable ASCII characters or a dot (`.`) fallback.
*   **Performance Cap**: Truncates output after 1024 rows to ensure rendering stays fast.

### Completed Plugins

#### A. Integer Viewer (`int`)
*   **Path**: [plugins/int-viewer/index.ts](../../../plugins/int-viewer/index.ts)
*   **Behavior**:
    *   Parses input as a little-endian `u32` array (4 bytes per value).
    *   Renders a clear tabular view: Index | Decimal | Hex | Binary.
    *   Integrates an interactive signed vs. unsigned representation toggle via Extism configuration:
        ```typescript
        let signed = Config.get("signed") == "true";
        ```

#### B. Text Viewer (`txt`)
*   **Path**: [plugins/txt-viewer/index.ts](../../../plugins/txt-viewer/index.ts)
*   **Behavior**:
    *   Performs auto-detection between ASCII and UTF-16LE null-terminated encodings.
    *   Because AssemblyScript's `String.fromCharCode` does not support variadic arguments for multi-byte Unicode chars, it decodes and appends them custom-coded code-unit by code-unit.
    *   Sanitizes text by escaping HTML special characters (`&`, `<`, `>`, `"`, `'`).
    *   Provides fallback to raw hex view if decoding fails.

#### C. Texture Viewer (`tex` / `btbl`)
*   **Path**: [plugins/tex-viewer/index.ts](../../../plugins/tex-viewer/index.ts)
*   **Behavior**:
    *   Detects if the resource is a plain Texture (`0x00070007` header sentinel) or a Bitmap Table (`btbl`).
    *   **Textures (`tex`)**: Reads counts, Flic entries, relocation data, and maps unknown structures.
    *   **Bitmap Tables (`btbl`)**: Parses bitmap table headers, image dimensions, formats (`A8R8G8B8`, `R5G6B5`, `R8G8B8`), and mipmaps.
    *   **BMP Preview Generation**: Decodes raw pixels into standard Microsoft BMP file structures and embeds them inline via Base64 strings.
        ```typescript
        const base64 = encodeBase64(bmp);
        preview = "<img src=\"data:image/bmp;base64," + base64 + "\" />";
        ```

#### D. Manifold Mesh Viewer (`mam`)
*   **Path**: [plugins/mam-viewer/index.ts](../../../plugins/mam-viewer/index.ts)
*   **Behavior**:
    *   Unpacks bounding box float fields (`bbox_min`, `bbox_max`).
    *   Exposes total vertex counts, face counts, and estimates data size layout versus actual OVL resource size.

#### E. Sound (`snd`) & Spline (`spl`) Viewers
*   **Sound**: Renders Wave format metadata structures (e.g. sample rates, channels, format codes).
*   **Spline**: Renders Bézier curves control points and spline segment metadata.

## 4. Build System & Developer Workflow

All plugins are developed in AssemblyScript and compiled to standalone `.wasm` targets.

### Compilation Config
Each plugin directory requires an [asconfig.json](../../../plugins/int-viewer/asconfig.json) mapping target configurations:
```json
{
  "targets": {
    "release": {
      "optimizeLevel": 3,
      "shrinkLevel": 0,
      "converge": false,
      "noAssert": true
    }
  }
}
```

### Compiler Script
The build execution is managed via [build-plugins.ts](../../../scripts/build-plugins.ts):
1.  Downloads/locates local shims for `@extism/as-pdk`.
2.  Invokes AssemblyScript's programmatic compiler API (`asc.main()`).
3.  Outputs optimized WASM binaries directly to `bin/plugins/<name>.wasm`.

To execute compilation:
```bash
deno run --allow-read --allow-write --allow-run=npx scripts/build-plugins.ts
```

### Deno Task Integrations
The [deno.json](../../../deno.json) file declares shortcut targets:
```json
"build:plugins": "deno run --allow-read --allow-write --allow-run=npx scripts/build-plugins.ts",
"check:plugins": "deno check plugins/int-viewer/index.ts plugins/txt-viewer/index.ts"
```

> [!NOTE]
> Standard TypeScript type checkers (like `deno check`) will report errors for AssemblyScript-specific types (`u32`, `i32`, `f32`, `bool`, etc.). This is expected behavior; plugins must be compiled using `asc` rather than standard TS checks.

## 5. Verification Checklist

Always test changes using these steps:
1.  Run the build command: `deno task build:plugins`.
2.  Ensure compiled `.wasm` targets are output to `bin/plugins/`.
3.  Load the OVL Dumper, select resource nodes of the targeted type, and verify HTML output in the content panel.
4.  Run unit tests to catch regression issues:
    ```bash
    make test
    ```
